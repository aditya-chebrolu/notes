| Strategy                         | Where ID is generated                         | 1 server + 1 DB | N servers + 1 DB             | 1 server + N DBs           | N servers + N DBs | Main issue                                           | Verdict                                          |
| -------------------------------- | --------------------------------------------- | --------------- | ---------------------------- | -------------------------- | ----------------- | ---------------------------------------------------- | ------------------------------------------------ |
| **DB Auto Increment**            | Database                                      | Excellent       | Works, but DB bottleneck     | Poor                       | Poor              | Hard to keep global uniqueness across DBs            | Best for small/simple systems                    |
| **Single Counter Service**       | Central ID service                            | Good            | Good, but central bottleneck | Fair                       | Fair              | ID service becomes SPOF/bottleneck unless replicated | Better than DB auto increment, still centralized |
| **DB Range Allocation**          | DB allocates ranges, server generates locally | Good            | Very good                    | Good with careful planning | Good              | Range coordination, wasted IDs on crash              | Strong practical scaling choice                  |
| **Snowflake-style ID**           | Application server / ID worker                | Good            | Excellent                    | Excellent                  | Excellent         | Clock drift, machine ID management                   | Best distributed choice overall                  |
| **Random String**                | Application server                            | Good            | Good                         | Fair                       | Fair to hard      | Collision detection/retries                          | Simple, but collision management grows at scale  |
| **Hash of Long URL**             | Application server                            | Good            | Good                         | Good                       | Good              | Hash collisions, inflexible product behavior         | Useful only if deterministic mapping is desired  |
| **Timestamp + Random**           | Application server                            | Good            | Good                         | Good                       | Good              | Can still collide under heavy concurrency            | Lightweight distributed option                   |
| **UUID + Base62**                | Application server                            | Good            | Excellent                    | Excellent                  | Excellent         | IDs are longer than ideal                            | Great for uniqueness, not great for short links  |
| **Shard Prefix + Local Counter** | Server/shard                                  | Good            | Very good                    | Very good                  | Excellent         | Operational complexity during rebalancing            | Strong choice for sharded systems                |
