| **Technique**                            | **Advantages**                                                                                                     | **Disadvantages**                                                                                                    | **Used In / Best For**                                                                                         |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| **🔹 Range-Based Sharding**              | • Simple to implement<br>• Efficient for range queries (BETWEEN, >, <)<br>• Data locality is preserved             | • Hotspot on high-value ranges (e.g., latest data)<br>• Uneven distribution if data is skewed<br>• Hard to rebalance | • Time-series databases (with caution)<br>• OLAP workloads<br>• Systems with predictable key distribution      |
| **🔹 Hash-Based (Modulo) Sharding**      | • Very even data distribution<br>• Simple formula (hash % N)<br>• Good for write-heavy workloads                   | • Massive rebalancing when adding/removing shards<br>• No support for range queries<br>• Poor flexibility            | • Small-to-medium distributed systems<br>• Stateless services<br>• Early-stage scalable architectures          |
| **🔹 Consistent Hashing**                | • Minimal data movement on scaling<br>• Highly scalable<br>• Works well with dynamic clusters                      | • More complex implementation<br>• Requires virtual nodes for balance<br>• Debugging is harder                       | • Cassandra, Dynamo-style systems<br>• Distributed caches (Redis, Memcached)<br>• Microservices routing layers |
| **🔹 Directory-Based Sharding**          | • Full control over data placement<br>• Easy to rebalance dynamically<br>• Supports custom sharding logic          | • Lookup table overhead<br>• Extra network hop<br>• Mapping table can become bottleneck                              | • Multi-tenant SaaS systems<br>• Systems needing custom routing logic<br>• Enterprise architectures            |
| **🔹 Geo-Based Sharding**                | • Low latency (data close to users)<br>• Helps with legal/compliance (GDPR)<br>• Natural partitioning              | • Uneven load across regions<br>• Cross-region joins are expensive<br>• Complex replication strategies needed        | • Global apps (Netflix, Uber)<br>• Region-specific services<br>• Compliance-heavy systems                      |
| **🔹 Entity-Based (Key-Based) Sharding** | • Keeps related data together<br>• Reduces cross-shard joins<br>• Simple mental model                              | • Hotspots for heavy users/tenants<br>• Load imbalance possible<br>• Hard to split large entities                    | • Social media platforms<br>• Multi-tenant SaaS apps<br>• User-centric architectures                           |
| **🔹 Time-Based Sharding**               | • Easy data lifecycle management (archive/delete)<br>• Great for sequential writes<br>• Works well for logs/events | • Recent shard becomes hotspot<br>• Queries across time hit multiple shards<br>• Requires retention strategy         | • Logging systems (ELK, Splunk)<br>• Analytics pipelines<br>• Event-driven architectures                       |
| **🔹 Hybrid Sharding**                   | • Combines strengths of multiple strategies<br>• Handles hotspots better<br>• Highly flexible and scalable         | • High design complexity<br>• Operational overhead<br>• Harder to debug and maintain                                 | • Large-scale systems (Google, Amazon, Meta)<br>• High-traffic platforms<br>• Systems with mixed workloads     |
| **🔹 Dynamic / Auto-Sharding**           | • Automatic balancing<br>• Minimal manual intervention<br>• Built-in scaling                                       | • Less control over placement<br>• Depends on DB internals<br>• Can have unpredictable behavior                      | • MongoDB, Vitess<br>• CockroachDB, YugabyteDB<br>• Cloud-native distributed databases                         |
